**Release:** v1.9.10
**Date:** 2026-05-15
**Scope:** Pre-v2.0 stratified validation. Six cloud strata each
with ~10 publicly-reproducible synthetic fixtures (60 stratum
fixtures + 19 base v1.9.9 fixtures = 79 total). Bayesian network

### Reading the per-stratum results

**Multi-cloud rollup behaviour by stratum:**
- PaaS fires multi-cloud most often (70%) because Vercel /
  Netlify / Cloudflare Pages fixtures legitimately span two cloud
  vendors (the PaaS provider + Cloudflare DNS).
- SSE/SASE never fires multi-cloud (0%): the SSE provider is a
  security service, not a cloud vendor in the rollup map. SSE
  fixtures with one true cloud vendor (e.g., AWS Route 53) cannot
  cross the >= 2 distinct vendor threshold. This is the right
  behaviour and matches how operators describe SSE deployments.
- Oracle / Alibaba / Azure rates are low (10-22%) because most
  fixtures in those strata are intentionally single-cloud
  (the stratum is the cloud).
- GCP fires more often (33%) because the GCP stratum includes
  multi-cloud variants (`stratum_gcp_dual_with_aws`,
  `stratum_gcp_idp_heavy` with Okta).

**Ceiling firing rate is high across stratum fixtures (78-100%)**
because the synthetic fixtures are intentionally weighted toward
sparse profiles (multi-domain + few categorized service families +
few surface attributions). On a real-world balanced corpus the
ceiling firing rate would be substantially lower.

The ceiling rate on the v1.9.9 baseline corpus (60%) is closer to
what a balanced operational corpus would show. The stratum
fixtures' high ceiling rate is a synthetic-corpus artefact, not a
production calibration concern.

## Honest framing

This is synthetic-corpus validation, not real-corpus validation. A
rigorous review correctly notes that:

1. The synthetic fixtures are designed by the same person who wrote
   the trigger heuristics, so they are biased toward demonstrating
   the surfaces.
2. The per-stratum patterns reflect intentional fixture design, not
   patterns observed in real data.

The corpus aggregator at `validation/corpus_aggregator.py` is ready
to run against the gitignored private corpus. The maintainer's
real-corpus run (whenever it happens) will produce per-stratum
firing rates that are the true measurement; the synthetic results
above are the publicly-reproducible specification audit.

The v1.9.10 ship line is honest: "we built the harness, ran it on
synthetic data we authored, the synthetic data behaves as
designed." Production-fidelity validation waits on the real-corpus
run, which is documented as standing work in
`validation/invariant_audit.md` item 1.
