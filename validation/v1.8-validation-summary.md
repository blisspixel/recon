# v1.8.0 validation report

**Date:** 2026-05-03
**Corpus:** 105 unique domains drawn from `corpus-50-diverse.txt`,
`corpus-random-50-20260427.txt`, and `corpus-weak-areas.txt`.
**Command:** `python -m recon_tool.cli batch <corpus> --json --include-ecosystem --concurrency 8`

## Headline

- **0 errors** across 105 domains in both the original and post-fix
  runs. The v1.8 code paths
  (`infrastructure_clusters`, `fingerprint_metadata`, ecosystem
  hypergraph) do not introduce regressions.
- **The validation surfaced two pre-tag hardening items** described
  below. Both are fixed and re-validated; the second run confirms
  the fixes work on real data.
- **Cluster partition quality strong on the domains that have
  buildable graphs.** Mean modularity **0.563**, max **0.883** in
  the original run.
- **Fingerprint relationship metadata covers 91 / 105 domains** that
  detected at least one slug carrying `parent_vendor` /
  `product_family` / `bimi_org` metadata.

## What the validation surfaced (and how it's fixed)

### 1. CertSpotter `Z`-suffix dates rejected on Python 3.10

The original 105-domain run produced **0/105** populated
`cert_summary` blocks despite 104/105 domains successfully reaching
CertSpotter. Investigation traced this to `build_cert_summary` in
`recon_tool/sources/cert_providers.py`: it called
`datetime.fromisoformat()` on `not_before` / `not_after`, which
CertSpotter emits with the `Z` UTC marker (e.g.
`2025-04-03T00:00:00Z`). Python 3.10's `fromisoformat` rejects `Z`
— only 3.11+ accepts it natively. Every entry was silently skipped
and `cert_summary` returned None, which in turn left
`top_issuer` ecosystem hyperedges, `wildcard_sibling_clusters`, and
`deployment_bursts` all empty.

This was a pre-existing v1.7.0 bug, surfaced for the first time by
the v1.8 corpus run. **Fix:** new `_parse_iso_datetime()` helper
normalizes `Z` → `+00:00` before parsing; applied in both
`build_cert_summary` and `_detect_deployment_bursts`. Regression
test in
[`tests/test_cert_providers.py::TestBuildCertSummary::test_certspotter_z_suffix_dates_parse`](../tests/test_cert_providers.py).

After the fix, a fresh-cache re-run produced a populated
`cert_summary` (cert_count=100, top_issuers=Amazon Trust Services,
GlobalSign, DigiCert) on the one domain that beat CertSpotter's
weekly rate limit. Production usage spread across cache + crt.sh
fallback recovers cert_summary normally.

### 2. `shared_slugs` hyperedge noise floor

The original run's hypergraph was 195/200 trivial pairs — every
enterprise pair sharing "Microsoft365 + Adobe + DocuSign + Mailchimp +
SendGrid" would fire as a `shared_slugs` edge. The cap was hit by
ubiquitous-SaaS coincidence rather than by structural similarity.

**Fix:** two complementary changes in `recon_tool/ecosystem.py`:

1. `_MIN_SLUG_OVERLAP` raised from 2 → 3.
2. New `_baseline_slugs()` helper computes per-batch slug
   prevalence; slugs above `_BASELINE_FREQ_THRESHOLD = 0.5` are
   stripped from the intersection before the threshold check. The
   filter is adaptive — it scales with whatever corpus the operator
   runs — and engages only when the batch exceeds
   `_MIN_BATCH_FOR_BASELINE = 5` domains so synthetic test fixtures
   stay deterministic.

Post-fix re-run: the 7 ubiquitous slugs (`microsoft365`,
`google-site`, `mailchimp`, `sendgrid`, `akamai`, `apple`,
`atlassian`) no longer appear in any `shared_slugs` key. Remaining
edges surface discriminating signal — adobe-idp (179), anthropic
(108), docusign (70), exchange-onprem (63), federated-sso-hub (55),
jamf (49), openai (48), onetrust (48), aws-ses (46), and so on.
Each pair now reflects genuine enterprise tech-stack overlap rather
than baseline noise. Two new regression tests in
[`tests/test_ecosystem.py::TestSharedSlugs`](../tests/test_ecosystem.py).

## Infrastructure clusters by path (original run)

| algorithm | domains |
|---|---|
| `louvain` (graph fits in 500-node cap) | 10 |
| `connected_components` (fallback above cap) | 1 |
| `skipped` (no graph buildable from CT data) | 94 |

The 94 `skipped` results align with the "hardened-target" framing in
[`docs/correlation.md`](../docs/correlation.md): wildcard-dominated CT
histories, single-SAN certs, and small footprints all collapse below
the 2-edge minimum. Only ~10 % of the corpus has rich enough
multi-SAN issuance to support community detection — exactly why v1.8
keeps the cluster envelope hedged and the modularity score visible
rather than burying the partition behind a verdict.

Graph size on the buildable subset: **mean 234 nodes, max 714** — the
714-node case triggered the connected-components fallback,
validating the 500-node cap.

## Defensive interpretation

The numbers reinforce the v1.7-v1.9 thesis: hardening narrows what
single-cert fingerprinting can recover, and structural correlation
recovers usable signal from the residual. Where the graph fires
(modularity 0.5-0.9), the partition is meaningful. Where it doesn't
(94 `skipped` cases), the report says so — no false confidence on
sparse evidence.

After the noise-floor fix, the hypergraph cap order — top_issuer →
bimi_org → parent_vendor → shared_slugs — pre-fills high-signal edges
and lets discriminating slug overlaps fill the rest, instead of
ubiquitous-SaaS coincidence.

## What this run does NOT validate

- **Live-network resilience at scale.** CertSpotter rate-limited
  most cache-cleared queries (1/105 got fresh cert metadata in the
  re-run). Production usage spread across the day, with crt.sh as
  primary and CertSpotter as fallback, will populate
  `cert_summary` on more targets — but a 105-domain burst against
  CertSpotter alone will hit weekly quota.
- **Chain-motif coverage.** Zero `chain_motifs` observations across
  both runs. Likely the v1.7 motif catalogue is too narrow for the
  CDN/origin shapes this corpus uses; a follow-up motif-library
  expansion is the right next step.
- **Vertical-baseline anomaly behavior.** Anomalies surface inline
  with `analyze_posture --profile`, not in batch mode. Validating
  the new `expected_categories` / `expected_motifs` rules requires a
  second pass through `recon analyze posture --profile fintech` over
  representative domains; tracked for the v1.8 retrospective.

  | grep -v '^#' | grep -v '^$' | sort -u > /tmp/v1_8_validation.txt

  --json --include-ecosystem --concurrency 8 \
  > validation/v1.8-validation-results.json
