# v2.0 release notes draft

This file stages the v2.0 release notes for the lock ceremony.
Until the ceremony, it lives here so it does not collide with the
`## [X.Y.Z]` section check in `scripts/release.py`. At lock time,
the maintainer moves the section below into `CHANGELOG.md` as
`## [2.0.0] - YYYY-MM-DD`, populates the date, and runs the
release script.

The draft is inventory-style, not feature-list. v2.0 ships no
new behavior; it locks the surface that the v1.9.x bridge
sequence validated.

---

## [2.0.0] - YYYY-MM-DD (DRAFT — do not move into CHANGELOG.md until lock)

**v2.0 schema-lock-and-tag ceremony.** Locks the public surface
that the v1.9.x bridge sequence validated. No new features, no
new fingerprints, no schema additions beyond the experimental →
stable promotions listed below. Every surface in
`docs/stability.md` carries the **Stable** guarantee from this
tag forward: breaking it requires a major version bump and a
deprecation window in the prior minor release.

For the validation evidence behind the lock, see
`validation/v2.0-validation-summary.md`. For the corpus-firing
truth-of-record, see `validation/v2.0-corpus-run.md`. For the
security-audit closure trail, see
`docs/security-audit-resolutions.md`.

Each surface below shipped pre-v2.0 with an `EXPERIMENTAL` label.

label is removed. See the schema-lock disposition table in

### Schema lock

- `docs/recon-schema.json` description bumped from "Stable v1.0
  contract" to "Stable v2.0 contract." Field set, types, and
  semantics unchanged from v1.9.14 except for the
  experimental → stable promotions above.
- All ~45 stable JSON fields previously covered by the v1.0
  contract remain covered; the Bayesian fusion fields
  (`slug_confidences`, `posterior_observations`) and the v1.7 /
  v1.8 structural fields move into the stable set per the table
  above.
- The data-file schemas (`fingerprints`, `signals`, `profiles`,
  per-domain CT cache, TenantInfo result cache,
  `RECON_CONFIG_DIR` environment variable) remain stable as
  documented in `docs/stability.md`.

### MCP tools

All 17 MCP tools listed in `docs/stability.md` remain stable.
The two MCP fusion tools (`get_posteriors`, `explain_dag`) move
from experimental to stable per the promotion table above.

### CLI commands and flags

Every CLI command and flag listed as **Stable** in
`docs/stability.md` (as of v1.9.14) carries that guarantee
forward. The two CLI fusion flags (`--fusion`, `--explain-dag`)
move from experimental to stable per the promotion table above.

### Python support

Tested range: Python 3.10 through 3.13, unchanged from v1.9.14.

### Bayesian network

The v1.9.10 stability disposition table holds at v2.0. All 9
nodes are `stable`; `okta_idp` is kept with the corpus-exposure
caveat recorded in `recon_tool/data/bayesian_network.yaml`. No
topology change, no CPT change since v1.9.6.

### Security

No new security work in v2.0. The May-2026 audit cycle closed
through v1.9.4, v1.9.9, v1.9.13, and v1.9.14 per
`docs/security-audit-resolutions.md`. Scanner re-flags against
pre-closure commits are resolved in favor of the closure record;
this is the project's authoritative disposition policy.
